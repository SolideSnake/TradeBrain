# TradeBrain MVP 功能清单

## 文档信息

- 版本：v0.3
- 状态：MVP v1 已完成
- 更新日期：2026-04-17
- 适用范围：TradeBrain 当前 MVP
- 文档目的：作为第一版 MVP 的范围定义与验收基线，避免旧的“任务中心”设计继续干扰实现

## 1. MVP 目标

TradeBrain 当前 MVP 只做一件事：

建立一个可本地运行的辅助交易工作台，让用户能够在不持续盯盘的情况下，看到关注标的、查看关键指标、获取结构化估值状态，并在关键状态变化时收到 Telegram 预警。

MVP 不追求完整交易系统，不追求复杂策略系统，也不追求 AI 能力，而是优先把以下闭环跑通：

`标的池 -> 盈透数据 -> 统一快照 -> 指标 -> 状态 -> 预警 -> 看板`

## 1.1 当前结论

截至 `2026-04-17`，本文件定义的 7 项 MVP 范围已经全部落地，第一版 MVP 结束。

当前已完成闭环：

- watchlist 管理
- IBKR 只读接入
- `CanonicalSnapshot`
- 核心指标
- PEG 驱动的估值状态
- 状态变化记录
- Telegram 预警
- Web 看板与 Settings 配置页

## 2. MVP 范围

MVP 当前严格只做以下 7 件事。

### 2.1 Watchlist 与分组管理

系统必须支持：

- 手动维护关注标的列表
- 为标的设置分组
- 标记哪些标的启用监控
- 标记哪些标的属于已持仓或观察池

第一版最低字段：

- 代码
- 名称
- 市场
- 资产类型
- 分组
- 是否启用监控
- 备注

### 2.2 IBKR 只读接入

系统必须支持通过 IBKR 读取：

- 账户摘要
- 当前持仓
- 当前持仓成本
- 美股或已关注标的行情

第一版原则：

- 只读
- 不下单
- 不修改订单

第一版最低可接受结果：

- 能成功连接 TWS 或 IB Gateway
- 能拉到账户和持仓
- 能拉到关注标的行情

### 2.3 CanonicalSnapshot

系统必须生成统一快照对象，将以下数据合并在一起：

- 标的配置
- 最新行情
- 最新持仓
- watchlist 配置

统一快照是以下模块的唯一输入来源：

- 指标计算
- 状态判断
- 预警生成
- 看板读模型

第一版必须避免：

- 页面自己拼装原始数据
- 各模块自行重复读取 IBKR 原始数据

### 2.4 核心指标

MVP 只计算最关键的指标，不做复杂指标系统。

第一版必须支持：

- 当前价格
- 当日涨跌幅 %
- 距 52 周高点回撤 %
- 距阶段高点回撤 %
- 持仓盈亏 %
- `PE`
- 盈利增长率 %
- `PEG`

第一版不做：

- 通用脚本化指标编辑器
- 高复杂度自定义表达式系统
- 交易执行参数计算

### 2.5 基础状态引擎

MVP 当前只做估值状态，不做执行状态。

第一版统一状态只有：

- `低估`
- `合理`
- `高估`

状态引擎输入：

- `CanonicalSnapshot`
- 核心指标
- `PEG` 估值结果

状态引擎输出至少包括：

- 当前状态
- 上一次状态
- 是否发生变化
- 状态更新时间

第一版不做：

- 买入 / 卖出 / 持有待办
- 止盈止损任务
- 复杂多策略并发冲突处理
- 高级策略 DSL

### 2.6 Telegram 预警

系统必须支持通过 Telegram Bot 发送关键提醒。

第一版只发送以下事件：

- 标的从 `低估 / 合理 / 高估` 中任一状态切换到另一状态
- Telegram 配置存在但发送失败
- IBKR 同步失败

预警消息最低字段：

- 标的代码或名称
- 当前状态
- 上一个状态
- `PEG`
- 时间

第一版必须支持：

- 通过前端配置 Telegram
- 后端本地保存配置
- 预警发送历史记录

### 2.7 Web 看板

MVP 前端当前只做 5 个核心页面：

1. 首页总览
2. 监控列表页
3. 预警页
4. 账户持仓页
5. 设置页

#### 首页总览

必须展示：

- 系统状态
- broker 状态
- 监控标的数量
- 账户摘要

#### 监控列表页

必须展示：

- 代码
- 名称
- 当前价格
- 涨跌幅
- 持仓状态
- 持仓盈亏
- 距 52 周高点回撤
- 距阶段高点回撤
- `PEG`
- 当前状态
- 状态变化

#### 预警页

必须展示：

- 最近预警记录
- 发送状态
- 预警内容
- 错误信息

#### 账户持仓页

必须展示：

- 账户净值
- 可用现金
- 当前持仓
- 未实现盈亏

#### 设置页

必须展示：

- Telegram 是否已配置
- 当前配置来源
- Bot Token 是否已保存
- Chat ID

第一版不做：

- 任务中心页
- 复杂图表工作台
- 多标签高级分析页
- 复杂拖拽式布局编辑器

## 3. 明确不做的内容

MVP 明确排除以下内容：

- 自动下单
- 回测系统
- 多用户系统
- 云同步
- 复杂策略 DSL
- AI 自动生成交易建议主流程
- 高级图表分析器
- 多券商接入
- 任务中心

## 4. MVP 数据流

MVP 固定按如下顺序工作：

1. 读取 watchlist 与配置
2. 从 IBKR 同步账户、持仓、行情
3. 构建 `CanonicalSnapshot`
4. 计算核心指标
5. 执行基础状态判断
6. 生成或发送 Telegram 预警
7. 投影为看板读模型
8. 页面读取结果展示

## 5. MVP 页面外行为

除了页面展示，MVP 还必须有以下系统行为：

- 定时刷新 IBKR 数据
- 定时或事件驱动刷新状态
- 记录预警发送历史
- 记录系统同步状态

## 6. MVP 验收标准

以下场景全部打通，即可视为当前 MVP 完成。

当前结果：以上场景已完成，MVP v1 已验收关闭。

### 场景 1：监控启动

- 用户配置好 IBKR 和 Telegram
- 用户添加一批关注标的
- 系统能够成功显示这些标的的基础监控信息

### 场景 2：持仓联动

- 系统能从 IBKR 读取持仓
- 页面能正确显示成本价、持仓盈亏和当前状态

### 场景 3：状态变化

- 某标的行情或基本面变化后，系统能够刷新指标
- 系统能够将状态更新为 `低估 / 合理 / 高估` 之一
- 系统能够记录状态前后变化

### 场景 4：Telegram 提醒

- 关键状态变化时，Telegram 能收到提醒
- 页面能看到发送历史
- 发送失败时能看到错误信息

### 场景 5：系统异常可见

- IBKR 断开时，页面能看到同步异常
- 系统能生成异常预警记录

## 7. 第一版实现顺序

MVP 当前实现顺序固定如下，且已经按该顺序完成：

1. Watchlist 与分组管理
2. IBKR 只读接入
3. `CanonicalSnapshot`
4. 核心指标
5. 基础状态引擎
6. Telegram 预警
7. Web 看板

说明：

- 这个顺序不是随意排列，而是按依赖关系排序。
- 任务中心已经移出当前 MVP，避免继续扩项。

## 8. 与模块设计文档的关系

本文件只定义当前 MVP “做什么”，不定义详细模块内部实现。

模块边界和目录结构以 [module-design.md](D:\code\TradeBrain\docs\module-design.md) 为准。

更长期的设计文档中仍可能保留“任务中心”作为后续能力，但它不属于当前 MVP 范围。

## 9. MVP 结束后的下一步

第一版 MVP 完成后，下一阶段优先处理以下事项：

1. 验证 live 模式下 PEG / 基本面字段的稳定性与覆盖率
2. 完成真实 Telegram 联调与日常使用回路验证
3. 视使用反馈决定是否增加 WebSocket 增量刷新
4. 再评估是否进入任务中心、图表增强或更多策略模块
