# TradeBrain MVP 功能清单

## 文档信息

- 版本：v0.1
- 状态：草案
- 更新日期：2026-04-09
- 适用范围：TradeBrain 第一版 MVP
- 文档目的：将第一版严格收敛为 8 个必须交付的功能项，作为实现阶段的直接范围边界

## 1. MVP 目标

TradeBrain MVP 只做一件事：

建立一个可本地运行的辅助交易工作台，让用户能够在不持续盯盘的情况下，看到关注标的、查看关键指标、获取结构化状态、生成待办任务，并在关键事件发生时收到 Telegram 预警。

MVP 不追求完整交易系统，不追求复杂策略系统，也不追求 AI 能力，而是优先把以下闭环跑通：

`标的池 -> 盈透数据 -> 统一快照 -> 指标 -> 状态 -> 任务 -> 预警 -> 看板`

## 2. MVP 范围

MVP 严格只做以下 8 件事。

### 2.1 Watchlist 与分组管理

系统必须支持：

- 手动维护关注标的列表
- 为标的设置分组
- 标记哪些标的启用监控
- 标记哪些标的属于已持仓或观察池

第一版最低字段：

- 代码
- 名称
- 市场
- 资产类型
- 分组
- 是否启用监控
- 备注

### 2.2 IBKR 只读接入

系统必须支持通过 IBKR 读取：

- 账户摘要
- 当前持仓
- 当前持仓成本
- 美股或已关注标的行情

第一版原则：

- 只读
- 不下单
- 不修改订单

第一版最低可接受结果：

- 能成功连接 TWS 或 IB Gateway
- 能拉到账户和持仓
- 能拉到关注标的行情

### 2.3 CanonicalSnapshot

系统必须生成统一快照对象，将以下数据合并在一起：

- 标的配置
- 最新行情
- 最新持仓
- watchlist 配置

统一快照是以下模块的唯一输入来源：

- 指标计算
- 状态判断
- 任务规划
- 看板读模型

第一版必须避免：

- 页面自己拼装原始数据
- 各模块自行重复读取 IBKR 原始数据

### 2.4 核心指标

MVP 只计算最关键的指标，不做复杂指标系统。

第一版必须支持：

- 当前价格
- 当日涨跌幅 %
- 振幅（高-低）
- 成交额

- 距 52 周高点回撤 %
- 距阶段高点回撤 %
- MA20

- 持仓盈亏 %
- 距止损位距离 %
- 距止盈位距离 %

第一版不做：

- 通用脚本化指标编辑器
- 高复杂度自定义表达式系统

### 2.5 基础状态引擎

MVP 只做 4 个统一状态：

- `观察`
- `接近触发`
- `待执行`
- `风险处理`

状态引擎输入：

- `CanonicalSnapshot`
- 核心指标
- 基础参数配置

状态引擎输出至少包括：

- 当前状态
- 触发原因
- 建议动作
- 风险等级

第一版不做：

- 复杂多策略并发冲突处理
- 高级策略 DSL

### 2.6 任务中心

系统必须把状态变化转成任务，而不是只显示建议文本。

第一版任务最低字段：

- 任务标题
- 关联标的
- 当前状态
- 建议动作
- 触发原因
- 优先级
- 更新时间

第一版任务状态最低支持：

- `待观察`
- `接近触发`
- `待执行`
- `已完成`
- `已忽略`
- `已失效`

系统必须支持：

- 自动生成任务
- 用户手动标记完成
- 用户手动忽略任务

### 2.7 Telegram 预警

系统必须支持通过 Telegram Bot 发送关键提醒。

第一版只发送以下事件：

- 标的从 `观察` 进入 `接近触发`
- 标的从 `接近触发` 进入 `待执行`
- 标的进入 `风险处理`
- IBKR 同步失败

预警消息最低字段：

- 标的代码或名称
- 当前价格
- 当前状态
- 建议动作
- 触发原因
- 时间

第一版必须支持：

- 去重
- 节流
- 测试发送

### 2.8 Web 看板

MVP 前端只做 4 个核心页面：

1. 首页总览
2. 监控列表页
3. 任务中心页
4. 账户持仓页

#### 首页总览

必须展示：

- 今日关键任务
- 高优先级预警
- 系统同步状态
- 已持仓摘要

#### 监控列表页

必须展示：

- 代码
- 名称
- 当前价格
- 涨跌幅
- 持仓状态
- 成本价
- 持仓盈亏
- 距 52 周高点回撤
- 距阶段高点回撤
- 距止损位
- 距止盈位
- 当前状态
- 当前任务

#### 任务中心页

必须展示：

- 所有未完成任务
- 任务状态
- 优先级
- 触发原因

#### 账户持仓页

必须展示：

- 账户净值
- 可用现金
- 当前持仓
- 未实现盈亏

第一版不做：

- 复杂图表工作台
- 多标签高级分析页
- 复杂拖拽式布局编辑器

## 3. 明确不做的内容

MVP 明确排除以下内容：

- 自动下单
- 回测系统
- 多用户系统
- 云同步
- 复杂策略 DSL
- AI 自动生成交易建议主流程
- 高级图表分析器
- 多券商接入

## 4. MVP 数据流

MVP 固定按如下顺序工作：

1. 读取 watchlist 与配置
2. 从 IBKR 同步账户、持仓、行情
3. 构建 `CanonicalSnapshot`
4. 计算核心指标
5. 执行基础状态判断
6. 生成或更新任务
7. 生成或发送 Telegram 预警
8. 投影为看板读模型
9. 页面读取结果展示

## 5. MVP 页面外行为

除了页面展示，MVP 还必须有以下系统行为：

- 定时刷新 IBKR 数据
- 定时或事件驱动刷新状态
- 记录任务变化历史
- 记录预警发送历史
- 记录系统同步状态

## 6. MVP 验收标准

以下场景全部打通，即可视为 MVP 完成：

### 场景 1：监控启动

- 用户配置好 IBKR 和 Telegram
- 用户添加一批关注标的
- 系统能够成功显示这些标的的基础监控信息

### 场景 2：持仓联动

- 系统能从 IBKR 读取持仓
- 页面能正确显示成本价、持仓盈亏和当前状态

### 场景 3：状态变化

- 某标的行情变化后，系统能够刷新指标
- 系统能够将状态更新为观察 / 接近触发 / 待执行 / 风险处理之一

### 场景 4：任务生成

- 状态变化后，系统能自动生成或更新任务
- 用户能在任务中心查看并手动处理任务

### 场景 5：Telegram 提醒

- 关键状态变化时，Telegram 能收到提醒
- 重复事件不会在短时间内无限刷屏

### 场景 6：系统异常可见

- IBKR 断开时，页面能看到同步异常
- 系统能生成异常预警记录

## 7. 第一版实现顺序

MVP 实现顺序固定如下：

1. Watchlist 与分组管理
2. IBKR 只读接入
3. `CanonicalSnapshot`
4. 核心指标
5. 基础状态引擎
6. 任务中心
7. Telegram 预警
8. Web 看板

说明：

- 这个顺序不是随意排列，而是按依赖关系排序。
- 只有当前一步稳定后，下一步实现才不会返工。
